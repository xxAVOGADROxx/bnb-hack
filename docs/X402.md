# x402 micropayments

The agent uses the x402 protocol on **both sides**: it pays for data it needs,
and it charges for data it produces. This is real, signed, on-chain activity —
not a README claim.

## The agent pays (consumer)

In a grey-zone decision — a near-miss setup under a conflicted regime — the agent
buys one premium technical-analysis pull per call, paid through x402. Payment is
cost-aware: it only pays when the answer could change the decision, so spend is
tied to genuine uncertainty rather than every cycle.

- Entry point: `agent/x402/premium.py` (the tie-break path).
- Settlement asset and venue follow the x402 facilitator; the signer reuses the
  in-memory key pattern (`agent/keys.py`) so the keystore stays the source of
  truth.

**Status (honest):** the pay path is implemented and proven end-to-end up to the
signature — TWAK detects the 402, selects the BSC route, broadcasts the one-time
approval and signs the payment. It is currently blocked by a TWAK transport bug:
CMC's MCP requires `Accept: application/json, text/event-stream`, and TWAK's paid
retry omits the event-stream part, so CMC returns HTTP 400 *before* settlement —
no funds move. The branch **degrades safely** (a blocked/failed call counts as
"no confirmation", so the agent stays out), and lights up with no code change
once TWAK sends the header. This is genuine sponsor-tooling feedback, not a
hidden gap.

## The agent charges (provider)

A built-in x402 V2 server sells a **catalog** of the agent's read-only data
products, one call at a time:

```bash
python -m agent.x402.server
```

- `GET /catalog` (free) lists the priced products and their price.
- `GET /leaderboard` — the field-wide competition ranking.
- `GET /posture` — the agent's current risk posture (high-water mark, baseline,
  recent snapshots and regime reads).
- `GET /report` — the latest periodic operations report.

Adding a product is one entry in the server's `PRODUCTS` table plus a producer
function reading an artefact the agent already generates — the
402/verify/settle rail is shared, so the offering grows with no new signing
surface.

- A request without payment receives a 402 challenge with the price and asset.
- The client pays with an EIP-3009 authorization; the server verifies the
  signature off-chain and submits settlement on-chain
  (`transferWithAuthorization` — the authorization nonce makes replays
  impossible).
- Validated end-to-end with a real self-payment: signed, verified, settled, and
  the data returned.

## Accounting

Both sides write to one append-only payments ledger (`agent/x402/ledger.py`,
`data/payments.jsonl`): the server records each settled **charge** (revenue) and
the premium branch records each **spend** (cost). The periodic report reads it,
so every operations report shows x402 revenue and spend next to trades, and the
operator can see whether the data service is self-funding.

## Alerts

Operational alerts are delivered to Telegram: executed and failed swaps, the
drawdown ladder, a dead data feed, and an hourly balance/return heartbeat. The
server now also alerts on each **settled x402 charge** (`💰 x402 charge …`), so
revenue lands in the same channel as trades. See
[`../deploy/WINDOW.md`](../deploy/WINDOW.md) for the full alert set.

## Further expansion

The same two primitives generalize further: more priced products (decision
traces, liquidity-sentinel readings, backtest specs), more paid inputs through
the cost-aware tie-break, and tiered or rate-limited access — each is a producer
entry or a tie-break call, not new protocol work.

# x402 micropayments

The agent uses the x402 protocol on **both sides**: it pays for data it needs,
and it charges for data it produces. This is real, signed, on-chain activity —
not a README claim.

## The agent pays (consumer)

In a grey-zone decision — a near-miss setup under a conflicted regime — the agent
buys one premium technical-analysis pull per call, paid through x402. Payment is
cost-aware: it only pays when the answer could change the decision, so spend is
tied to genuine uncertainty rather than every cycle.

- Entry point: `agent/x402/` (the tie-break path).
- Settlement asset and venue follow the x402 facilitator; the signer reuses the
  in-memory key pattern (`agent/keys.py`) so the keystore stays the source of
  truth.

## The agent charges (provider)

A built-in x402 V2 server sells the agent's live competition leaderboard, one
call at a time:

```bash
python -m agent.x402.server
```

- A request without payment receives a 402 challenge with the price and asset.
- The client pays with an EIP-3009 authorization; the server verifies the
  signature off-chain and submits settlement on-chain
  (`transferWithAuthorization` — the authorization nonce makes replays
  impossible).
- Validated end-to-end with a real self-payment: signed, verified, settled, and
  the data returned.

## Expansion paths

The same two primitives generalize cleanly:

- **More paid data products.** Any read-only artefact the agent already produces
  — per-token risk posture, decision traces, liquidity-sentinel readings,
  backtest specs — can be exposed as a priced x402 endpoint with no new signing
  surface.
- **More paid inputs.** Additional premium feeds (deeper history, alternative
  signals, inference) can be consumed through the same cost-aware tie-break, so
  the agent pays for an edge only when it is uncertain.
- **Usage-based access.** Rate-limited or tiered access to the leaderboard and
  reports, priced per call, turns the agent into a self-funding data service.

## Alerts

Operational alerts are delivered to Telegram: executed and failed swaps, the
drawdown ladder, a dead data feed, and an hourly balance/return heartbeat.
Payment events (a settled x402 charge, a premium pull) are natural additions to
the same channel, so the operator sees revenue and spend in the same place as
trades. See [`../deploy/WINDOW.md`](../deploy/WINDOW.md) for the current alert
set.
